## Fixed Bugs

### BUG-001 — Timezone comparison crash in `InactivityFilter`
**Severity:** High (runtime crash)
**File:** `backend/inactivity_filter/__init__.py` — `InactivityThreshold.is_inactive()`
**Status:** ✅ Fixed

**Description:**
`InactivityThreshold.is_inactive()` compared `last_message_time <= self.date` directly. Telegram returns timezone-aware datetimes (UTC). The threshold date is created from user-supplied strings and is timezone-naive. Python raises `TypeError: can't compare offset-naive and offset-aware datetimes` whenever a group with a Telegram-sourced timestamp was filtered.

**Fix:** Both sides are normalised to timezone-naive before comparison by stripping `tzinfo` if present.

**Regression test:** `smoke_test.py` — "BUG-001 regression: aware vs naive no crash"

### BUG-002 — Thread-safety: `save_groups` called outside the state lock
**Severity:** Medium (data race / inconsistent save)
**File:** `main.py` — `automation_worker()` and `_run_send()`, around the `successful_ids` block
**Status:** ✅ Fixed

**Description:**
After updating `g.last_message_time` inside `with app_state_lock:`, the call `persistence.save_groups([g.to_dict() for g in app_state.groups])` iterated `app_state.groups` *outside* the lock. Another thread (e.g. a Flask request handler) could modify the list between the lock release and the save, producing a corrupt or partially updated JSON file on disk.

This pattern existed in **two places**: `automation_worker()` and the `_run_send()` coroutine inside the broadcast handler. The initial fix only patched `automation_worker()`; the `_run_send()` instance was found by a follow-up code review and fixed separately.

**Fix:** In both locations, the snapshot `[g.to_dict() for g in app_state.groups]` is now taken *inside* the lock and passed to `save_groups` after releasing it.

### BUG-003 — `generate_session.py` crashes on import due to non-existent Telethon function
**Severity:** High (prevents session generation)
**File:** `generate_session.py`
**Status:** ✅ Fixed

**Description:**
The top-level import `from telethon.functions.account import ImportWebAuthorizationsLayer` fails on all current Telethon releases because this function does not exist in the public API. Python evaluates top-level imports before running any code, so the script crashed immediately with `ImportError` before `main()` was even called. The actual invocation was wrapped in `try/except` but the import was not.

**Fix:** Removed the import and the unused call block entirely.

### BUG-004 — `is_sending` flag permanently set on validation failure
**Severity:** Critical (denial-of-service on the broadcast endpoint)
**File:** `main.py` — `send_messages()` route handler
**Status:** ✅ Fixed

**Description:**
`app_state.is_sending = True` was set at the top of the handler, before input validation. If `groups_to_send` was empty or `message_template` was blank, the handler returned a 400 response **without ever resetting the flag**. All subsequent broadcast attempts returned `"Sending already in progress"` until the process was restarted.

**Fix:** Moved `app_state.is_sending = True` to after all validation checks pass. Added a second in-progress guard at the point of flag acquisition to keep the check atomic under concurrent requests.

**Regression tests:**
- `tests/test_message_sender.py::TestIsSendingFlagLeak::test_empty_message_does_not_lock_is_sending`
- `tests/test_message_sender.py::TestIsSendingFlagLeak::test_no_groups_does_not_lock_is_sending`
- `smoke_test.py` — "BUG-004 regression: is_sending flag leak"

### BUG-005 — `FloodWaitError` silently drops the group (no retry)
**Severity:** Medium (messages not sent to rate-limited groups)
**File:** `backend/message_sender/__init__.py` — `_send_single_message()`
**Status:** ✅ Fixed

**Description:**
When Telegram returned a `FloodWaitError`, the code slept for the required duration and then returned `RATE_LIMITED` status. The group was permanently skipped — no send was ever attempted after the wait.

**Fix:** After sleeping, the code retries `send_message` once:
- Retry succeeds → result is `SENT`
- Retry also raises → result is `FAILED`
- `_should_stop` was set during the wait → retry skipped, result is `RATE_LIMITED`

**Regression tests:**
- `tests/test_message_sender.py::TestFloodWaitRetry::test_flood_wait_retries_and_succeeds`
- `tests/test_message_sender.py::TestFloodWaitRetry::test_flood_wait_retry_fails_returns_failed`
- `tests/test_message_sender.py::TestFloodWaitRetry::test_flood_wait_during_stop_skips_retry`

### BUG-006 — `progress` property always reported 1.0 (or nonsense)
**Severity:** Low (broken UI progress bar)
**File:** `backend/message_sender/__init__.py` — `MessageSender.progress`
**Status:** ✅ Fixed

**Description:**
```python
# Old (wrong)
total = self._sent_count + self._failed_count
return total / len(self._results) if self._results else 0
```
`_sent_count + _failed_count` equals `len(self._results)` minus SKIPPED/RATE_LIMITED entries, so in the common case the ratio was always `1.0`. Even when not, the denominator tracked already-processed items rather than total groups to send, making the value meaningless as a progress indicator.

**Fix:** Added `_total_groups` field set at the start of `send_messages()`. Progress is now `len(self._results) / self._total_groups`, correctly representing the fraction of groups processed so far.

**Regression tests:**
- `tests/test_message_sender.py::TestProgressAndSummary::test_progress_zero_when_no_total`
- `tests/test_message_sender.py::TestProgressAndSummary::test_progress_reflects_processed_fraction`
- `tests/test_message_sender.py::TestProgressAndSummary::test_progress_full_when_all_processed`

### BUG-007 — `get_results_summary` "pending" counted SKIPPED/RATE_LIMITED as pending
**Severity:** Low (incorrect status display)
**File:** `backend/message_sender/__init__.py` — `get_results_summary()`
**Status:** ✅ Fixed

**Description:**
`pending = len(results) - sent - failed` yielded the count of SKIPPED/RATE_LIMITED results (already processed), not the number of groups not yet attempted. The UI would show non-zero "pending" even after automation fully completed.

**Fix:** `pending = max(0, _total_groups - len(results))` — groups that haven't been attempted yet. Uses `max(0, ...)` to guard against the impossible-but-safe case where results exceed total.

**Regression tests:**
- `tests/test_message_sender.py::TestProgressAndSummary::test_summary_pending_count`
- `tests/test_message_sender.py::TestProgressAndSummary::test_summary_pending_never_negative`

## Known Issues / Warnings (not yet fixed)

### WARN-001 — `from datetime import timedelta` inside a hot loop
**Severity:** Low (performance, not a bug)
**File:** `main.py` — `automation_worker()`, inside the `while True:` loop

Python caches module imports so this does not cause incorrect behaviour, but importing inside a loop adds unnecessary dictionary lookups on every iteration. The `timedelta` import should be moved to the top of the file alongside the existing `datetime`/`timezone` imports.

### WARN-002 — `Group.from_dict` produces timezone-naive datetimes
**Severity:** Low (mitigated by BUG-001 fix)
**File:** `backend/group_scanner/__init__.py` — `Group.from_dict()`

ISO strings stored by `Group.to_dict()` (e.g. `"2024-01-15T10:30:00"`) are loaded back without timezone info. Live-scanner events set `last_message_time = event.date`, which is timezone-aware (Telethon always uses UTC). This creates a mixed-awareness state in `app_state.groups`. BUG-001's fix prevents a crash at the comparison site, but the proper fix is to always store and load timezone-aware datetimes: append `+00:00` when serialising, parse with `timezone.utc` when deserialising.

**Smoke test confirms** the mitigated path does not crash (`smoke_test.py` — "WARN-002 regression").

### WARN-003 — `automation_worker` evaluates stale group snapshot
**Severity:** Low (over-messaging edge case)
**File:** `main.py` — `automation_worker()`

`current_groups = list(app_state.groups)` is captured once at the start of each worker iteration. If a new message arrives in a group *between* the snapshot and the send, the worker will still send to that group this cycle. The group's `last_message_time` is updated afterward, so the *next* cycle is correct. At most one superfluous message per cycle per affected group.

### WARN-004 — `config.session_file` does not update when `data_dir` is overridden at runtime
**Severity:** Low (configuration issue)
**File:** `config/__init__.py`

`session_file` is defined as a class-level default: `os.path.join(data_dir, "session.session")`. This evaluates at class definition time. If `data_dir` is later overridden on a `Config` instance, `session_file` will still point to the original path. Only affects code that programmatically changes `config.data_dir` after startup — not the normal deployment path.

### WARN-005 — `Scheduler` class is instantiated but never used
**Severity:** Info (dead code)
**File:** `backend/scheduler/__init__.py`, `main.py`

A full `Scheduler` class with daily/interval scheduling logic is defined and a global `scheduler` instance is created, but it is never wired to any callback in `main.py`. All actual scheduling is handled by `automation_worker()` (60-second polling) and `AdScheduler` (APScheduler cron). The `Scheduler` class is unused dead code that may confuse future maintainers.
