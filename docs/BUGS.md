# Bug Report & Fixes

Generated by code audit. All bugs below were found by static analysis, manual code review, and test execution.

---

## Fixed Bugs

### BUG-001 — Timezone comparison crash in `InactivityFilter`
**Severity:** High (runtime crash)
**File:** `backend/inactivity_filter/__init__.py` — `InactivityThreshold.is_inactive()`
**Status:** Fixed

**Description:**
`InactivityThreshold.is_inactive()` compared `last_message_time <= self.date` directly. Telegram returns timezone-aware datetimes (UTC). The threshold date is created from user-supplied strings and is timezone-naive. Python raises `TypeError: can't compare offset-naive and offset-aware datetimes` whenever a group with a Telegram-sourced timestamp was filtered.

**Fix:** Both sides are normalised to timezone-naive before comparison by stripping `tzinfo` if present.

---

### BUG-002 — Thread-safety: `save_groups` called outside the state lock
**Severity:** Medium (data race / inconsistent save)
**File:** `main.py` — `automation_worker()`, around the `successful_ids` block
**Status:** Fixed

**Description:**
After updating `g.last_message_time` inside `with app_state_lock:`, the call `persistence.save_groups([g.to_dict() for g in app_state.groups])` iterated `app_state.groups` *outside* the lock. Another thread (e.g. a Flask request handler) could modify the list between the lock release and the save, producing a corrupt or partially updated JSON file on disk.

**Fix:** The snapshot `[g.to_dict() for g in app_state.groups]` is now taken *inside* the lock and passed to `save_groups` after releasing it.

---

### BUG-003 — `generate_session.py` crashes on import due to non-existent Telethon function
**Severity:** High (prevents session generation)
**File:** `generate_session.py`, line 20
**Status:** Fixed

**Description:**
The top-level import `from telethon.functions.account import ImportWebAuthorizationsLayer` fails on all current Telethon releases because this function does not exist in the public API. Because Python evaluates top-level imports before running any code, the script crashed immediately with `ImportError` before `main()` was even called. The actual invocation was wrapped in `try/except` but the import was not.

**Fix:** Removed the import and the unused call block entirely.

## Known Issues / Warnings (not yet fixed)

### WARN-001 — `from datetime import timedelta` inside a hot loop
**Severity:** Low (performance, not a bug)
**File:** `main.py` — `automation_worker()`, inside the `while True:` loop

Python caches module imports so this does not cause incorrect behaviour, but importing inside a loop adds unnecessary dictionary lookups on every iteration. The `timedelta` import should be moved to the top of the file alongside the existing `datetime`/`timezone` imports.

---

### WARN-002 — `Group.from_dict` produces timezone-naive datetimes
**Severity:** Low (mitigated by BUG-001 fix)
**File:** `backend/group_scanner/__init__.py` — `Group.from_dict()`

ISO strings stored by `Group.to_dict()` (e.g. `"2024-01-15T10:30:00"`) are loaded back without timezone info. Live-scanner events set `last_message_time = event.date`, which is timezone-aware (Telethon always uses UTC-aware datetimes). This creates a mixed-awareness state in `app_state.groups`. BUG-001's fix prevents a crash, but the real fix is to always store and load timezone-aware datetimes (append `+00:00` when serialising, parse with `timezone.utc` when deserialising).

---

### WARN-003 — `automation_worker` evaluates stale group snapshot
**Severity:** Low (over-messaging edge case)
**File:** `main.py` — `automation_worker()`

`current_groups = list(app_state.groups)` is captured once at the start of each worker iteration. The inactivity check (`msg_time < threshold_time`) runs against this snapshot. If a new message arrives in a group *between* the snapshot and the send, the worker will still send a message to that group during this cycle. The group's `last_message_time` is updated afterward, so the *next* cycle will be correct, but one superfluous message may be sent per cycle per affected group.

---

### WARN-004 — `config.session_file` does not update when `data_dir` is overridden at runtime
**Severity:** Low (configuration issue)
**File:** `config/__init__.py`

`session_file` is defined as a class-level default: `os.path.join(data_dir, "session.session")`. This evaluates at class definition time. If `data_dir` is later overridden on a `Config` instance, `session_file` will still point to the original path. Affected only if code programmatically changes `config.data_dir` after startup.

---

### WARN-005 — `Scheduler` (`backend/scheduler/__init__.py`) is instantiated but never used
**Severity:** Info (dead code)
**File:** `backend/scheduler/__init__.py`, `main.py`

A full `Scheduler` class with daily/interval scheduling logic is defined and a global `scheduler` instance is created, but it is never wired up to any automation callback in `main.py`. All actual scheduling is handled by `automation_worker()` polling every 60 seconds and by `AdScheduler`. The `Scheduler` class is unused dead code that may confuse future maintainers.
